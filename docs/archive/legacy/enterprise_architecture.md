# EvoQuant: 企业级 AI 自主量化交易系统架构设计与演进方案

## 1. 核心愿景与系统定位 (Core Vision & System Positioning)

本项目旨在将最初的实验性框架（Toy Project）全面升级为**准工业级的 AI 自主量化投研平台**。系统的核心定位是构建一个高度自动化的“数字量化研究员团队”，它们在完全隔离、安全的沙箱环境中进行 24/7 的无间断量化研究（Alpha 挖掘、策略回测、风险评估）。

在这个生态中，**人类的角色将从“执行者/程序员”转变为“投资总监 (CIO)”**：人类负责提供宏观市场假设、设定风险偏好底线、审核由 AI 代理提交的策略报告，并做出最终的实盘部署决策。

## 2. 核心系统架构设计 (System Architecture Design)

系统将从单一脚本演进为基于微服务架构的分布式系统，以便于未来顺畅地部署到独立服务器集群上。

### 2.1 基础设施层：高度隔离的执行沙箱 (Secure Execution Sandbox)
作为一个由 LLM 驱动生成不可预知代码的系统，**沙箱是整个系统的生命线**。
*   **资源与网络隔离**：AI 生成的策略代码必须在严格限制了 CPU、内存和无外部网络访问权限的容器中执行，防止恶意代码（如数据窃取、系统破坏）或死循环耗尽服务器资源。
*   **统一数据接口**：沙箱内部暴露标准化的内部 API（如本地的离线聚合数据接口），禁止代码直接访问真实的互联网数据库或 API，消除未来函数泄露和安全隐患。
*   **执行状态监控**：主控系统实时监控沙箱的运行状态，捕获所有 `stdout/stderr` 异常，并将其清洗后作为结构化反馈传递回代理系统。

### 2.2 代理逻辑层：多智能体协作网络 (Multi-Agent Collaborative Network)
系统不再依赖单一的“全能”大模型，而是采用协同工作的 Agent 架构，各司其职：
*   **首席研究员代理 (Lead Researcher Agent)**：
    *   配置：由具备顶级逻辑推理能力的大模型驱动（如 DeepSeek-R1 / Claude 3.5 Sonnet）。
    *   **信息流与感知器 (Information Flow)**：Researcher 并非在真空中思考。我们需要为其构建“眼和耳”：
        *   **结构化数据源**：定时读取 Qlib 的大盘指数历史、板块动量排名。
        *   **非结构化数据源 (News Feed)**：接入类似 **金十数据 (Jin10)**、**东方财富** 的 RSS 或 API 接口，甚至通过爬虫获取当天的宏观政策摘要。系统会将这些新闻清洗为精炼的 Markdown 格式，每日定时喂给 Researcher。
    *   职责：融合多模态上下文（历史量化指标 + 最新财经文本），提出符合当前市场 Regime（牛市/熊市/主线轮动）的量化交易假设（Hypothesis）。
*   **量化工程师代理 (Quant Dev Agent - Autonomous CLI)**：
    *   配置：直接拉起 `opencode` (或 `claude code` / `Aider`) 等原生级别的终极代码助手 CLI。
    *   **安全沙箱 (Sandbox)**：为了防止 OpenCode 删库跑路或造成本地环境污染，我们将其实例化在一个**完全隔离的 Docker 容器**中。 LangGraph 宿主机通过 `oh-my-opencode` 接口向该容器内的 OpenCode 发送指令。容器内挂载了 Qlib 数据集，但没有外部网络访问权限（除了访问大模型 API）。
    *   职责：接收 Researcher 下发的《因子逻辑描述与回测任务单》，在 Docker 沙箱内自主完成 Qlib 策略代码的编写、Debug 和回测执行。最后将回测生成的报告返回给主系统。
*   **风控合规代理 (Risk Management & Compliance Agent)**：
    *   配置：专注于找茬和批判性思维的模型。
    *   职责：在代码执行前进行静态代码分析（是否存在未来函数？是否过度拟合？是否在非交易时间交易？），并在回测后分析性能指标的核心缺陷（如对极端行情的脆弱性）。

### 2.3 分层式智能体记忆系统 (Hierarchical Agent Memory System)
为了让智能体在无尽的市场循环中持续进化而不至于退化或重复踩坑，我们在微服务中引入三层记忆架构：
1.  **短期工作记忆 (Short-Term Working Memory)**：存储当前研报周期、当前对话线程的即时上下文（如最近 5 天的新闻池、当前正在 Debug 的完整错误堆栈）。任务结束后清空。
2.  **中期战术经验池 (Mid-Term Episodic Memory)**：基于向量数据库（如 ChromaDB/Milvus）存储的历史回测快照。当 Researcher 提出新策略时，主脑会通过 RAG 召回类似因子的历史执行结果，作为先验经验注入（例：“记忆告警：3周前测试的类似高频动量因子，在加入滑点后胜率狂跌 40%”）。
3.  **长期核心信条 (Long-Term Core Beliefs)**：系统运行的底线原则与教典。例如刚梳理出的《Qlib 完美数据转换实践（复权与停牌处理铁律）》，将作为 Long-Term Context 死死烙印在 Researcher 和 Coder 的 System Prompt 底部，不可篡改。

### 2.4 人机交互层 (Human-AI Interaction Layer)
*   **审批流系统 (Approval Workflow)**：AI 挖掘出夏普比率等指标达标的策略后，自动生成 Markdown 格式的“研报”，人类在前端 Dashboard 仅需点击“驳回（附带修改建议）”或“批准进入实盘模拟”。
*   **干预机制**：人类可以随时重置 Agent 的记忆上下文，或强制下发特定交易主题。

## 3. 核心技术栈选型决策 (Core Technology Stack Decision)

经过深入调研与讨论，我们确立了**“Qlib 作为投研大脑，轻量接口作为实盘手脚”**的解耦架构。

### 3.1 为什么选择 Microsoft Qlib？
1. **AI 擅长“写公式”，不擅长“盯盘”**：LLM 极其擅长逻辑演绎（如“寻找低估值且资金流入的股票”）并将其翻译为数学表达式（因子）。Qlib 的核心逻辑正是“因子挖掘 -> 机器学习打分 -> 排序组合”，这与 LLM 的能力模型完美契合。如果让 AI 直接处理 Tick 级订单簿，极易产生逻辑灾难。
2. **A 股截面多因子特性**：A 股为 T+1 市场且做空受限，当前最稳健的策略是“横截面多因子选股”（如中证 500/1000 增强）。Qlib 原生支持全市场数千只股票的高效截面打分与组合优化，远超传统事件驱动框架（如 VN.PY）在处理此类资产组合时的性能边界。

### 3.2 业界可借鉴的 AI 量化范式
*   **FinGPT (Confidence-Weighted Scoring)**：不仅让 AI 给出方向，更利用其文本阅读能力提取**情绪与置信度**。这可以作为 Qlib 中的一种全新非线性因子。
*   **nof1.ai Alpha Arena**：参考其 AI 交易竞技场的 Prompt 设计，规范化地将市场数据转化为 Context 送入模型。
*   **Qlib 因子变异算法 (Factor Mutation)**：让 AI 扮演“矿工”，读取现有的经典 Alpha158 因子，通过量价逻辑（量价背离、动量反转）基因重组出新的因子表达式，交由 Qlib 计算 IC/IR 自动淘汰劣质因子。

## 4. 落地路线图 (Action Plan: 从滑板车到自动驾驶汽车)

我们采用敏捷开发的“造车”哲学：不一开始就造所有零件，而是先造一个能滑的滑板车 (MVP)，然后再逐步升级为自行车、汽车。

### 第 1 阶段：物理底座闭环 (The "Skateboard" - 预计 1-2 周)
为了极速跑通闭环，我们**不**一开始就处理全市场 5000 只股票。
*   **微缩宇宙 (Toy Universe)**：使用 AKShare 获取“沪深300 (CSI 300)”的当前成分股名单作为演示数据集。
*   **数据管道 (Data Pipeline)**：下载这 300 只股票过去 5 年的日线复权数据，并转换为 Qlib 原生 `.bin` 格式。
*   **基线对齐 (Baseline)**：在不引入 AI 的前提下，手动跑通 Qlib 内置的 **Alpha158 + LightGBM** 流程，生成收益率、最大回撤等回测图表。这是系统的心跳测试。

### 第 2 阶段：构建核心 Agent 编排 (OpenCode 整合)（预计 2-3 周）
*   **工作流编排**：引入 **LangGraph**。由它作为系统的主控神经，负责生成具体的量化研究任务、拆解逻辑，并进行后续的结果审判。核心循环为：`Researcher提出因子假设` -> 下发给 `OpenCode` 编写并执行 Qlib 脚本 -> `Critic` 对执行结果（如回测的 Tear Sheet）进行反思评估。
*   **终极代码智能体 (`oh-my-opencode`)**：放弃手写脆弱的代码生成逻辑，**直接采用 `opencode.ai` 等原生的终极代码助手 CLI**。
    *   为了在 Python 进程 (LangGraph) 中顺滑地驱使这个终极 CLI，我们将研发一层适配器 **`oh-my-opencode`**。它负责：组装系统 Context、将 Researcher 的指令包装成规范的临时 Prompt 文件、非阻塞地拉起 `opencode` 进程以自动写代码并跑回测，最后将 `opencode` 留在磁盘上的控制台截断输出反馈回 LangGraph 主脑。

### 第 3 阶段：开发 CIO 审批主控台 (Human-AI UI)（预计 1 周）
*   **前端工程**：构建基于 **React + Vite** 的 Web Dashboard。
*   **审批逻辑**：当 AI Agent 在后台挖出夏普比率大于预设阈值的策略时，将相关回测图表、因子逻辑解释推送到控制台。人类作为 CIO 进行复核，点击“批准”则进入实盘观测池。

### 第 4 阶段：实盘模拟桥接 (Future Deployment)
*   **解耦执行**：避开对 VN.PY 底层的重度改造。每天收盘后（15:00 后），离线运行获批的 Qlib 策略，生成次日 (T+1) 的**目标股票池及买卖权重 (Target Weights)**。
*   **极简执行脚本**：编写一个无 AI 参与的确定性 Python 脚本，读取上述权重，通过券商提供的 **QMT/Ptrade** 量化终端接口，在次日开盘利用 TWAP/VWAP 算法完成自动化调仓。
